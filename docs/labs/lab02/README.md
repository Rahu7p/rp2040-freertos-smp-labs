# Activity 1 - Independent Producer and Consumer Tasks

---

## Objective

Create two FreeRTOS tasks executing on different processor cores. Although the **Producer** continuously generates data, the **Consumer** cannot access those values because no communication mechanism has been implemented yet.

The objective of this activity is to recognize that tasks executing on different processor cores require an explicit communication mechanism to exchange information.

---

## Procedure

1. Open the **Template Project**.

2. Create two FreeRTOS tasks named:

   - `ProducerTask`
   - `ConsumerTask`

3. Configure both tasks with:

   - The same priority.
   - An infinite execution loop.
   - A delay of **500 ms**.

4. Configure task affinity so that:

   - `ProducerTask` always executes on **Core 0**.
   - `ConsumerTask` always executes on **Core 1**.

5. The `ProducerTask` should periodically print an incrementing integer value.

Example:

```text
Producer (Core 0) -> 0
Producer (Core 0) -> 1
Producer (Core 0) -> 2
...
```

6. The `ConsumerTask` should periodically display the following placeholder:

```text
Consumer (Core 1) <- ???
```

The question marks represent a value that the Consumer cannot obtain because no communication mechanism has been implemented yet.

7. Do **not** use any communication mechanism.

Do **not** use:

- Queue
- Binary Semaphore
- Mutex
- Event Group

8. Start the scheduler.

---

## Example Output

```text
Producer (Core 0) -> 0
Consumer (Core 1) <- ???

Producer (Core 0) -> 1
Consumer (Core 1) <- ???

Producer (Core 0) -> 2
Consumer (Core 1) <- ???
```

Notice that the Producer continuously generates values while the Consumer has no mechanism to obtain them.

---

## Expected Behavior

After running the application:

- The Producer always executes on **Core 0**.
- The Consumer always executes on **Core 1**.
- The Producer continuously generates new values.
- The Consumer periodically displays the placeholder `???`.
- The Consumer cannot obtain the values generated by the Producer.
- Both tasks execute independently because no communication mechanism exists between them.

# Activity 2 - Queue Communication

---

## Objective

Modify the application developed in **Activity 1** so that the **Consumer**, executing on **Core 1**, is able to display the values generated by the **Producer**, executing on **Core 0**.

The objective of this activity is to discover how a communication mechanism enables tasks running on different processor cores to exchange information safely.

---

## Procedure

Continue using the application developed in **Activity 1**.

1. The Producer already generates an incrementing integer value every 500 ms.

2. The Consumer currently displays:

```text
Consumer (Core 1) <- ???
```

Your objective is to replace the question marks with the values generated by the Producer.

3. Based on the concepts introduced in **Chapter 2**, identify the appropriate FreeRTOS communication mechanism.

> **Hint:** Use a **FreeRTOS Queue**.

4. Create a Queue capable of storing integer values.

5. Modify the `ProducerTask` so that every generated value is sent to the Queue.

6. Modify the `ConsumerTask` so that it waits for a value from the Queue and displays it after reception.

7. Keep the same task affinity configuration:

- `ProducerTask` executes on **Core 0**.
- `ConsumerTask` executes on **Core 1**.

8. Build and execute the application.

---

## Example Output

```text
Producer (Core 0) -> 0
Consumer (Core 1) <- 0

Producer (Core 0) -> 1
Consumer (Core 1) <- 1

Producer (Core 0) -> 2
Consumer (Core 1) <- 2
```

Notice that the values generated on **Core 0** are successfully received by the task executing on **Core 1**.

> [!NOTE]
>
> A FreeRTOS Queue behaves identically whether the communicating tasks execute on the same processor core or on different processor cores. The FreeRTOS SMP kernel automatically guarantees safe concurrent access to the Queue.

---

## Expected Behavior

After running the application:

- The Producer always executes on **Core 0**.
- The Consumer always executes on **Core 1**.
- Every value generated by the Producer is eventually received by the Consumer.
- The Producer and Consumer do not necessarily alternate their execution.
- The Queue preserves the order of the transmitted values (FIFO).

# Activity 3 - Multiple Producers

---

## Objective

Extend the application developed in **Activity 2** by introducing a second Producer task.

The objective of this activity is to observe how multiple tasks executing on different processor cores can safely share the same FreeRTOS Queue while a single Consumer receives all transmitted data.

---

## Procedure

Continue using the application developed in **Activity 2**.

1. Create a second Producer task named `ProducerTaskB`.

2. Configure task affinity as follows:

- `ProducerTaskA` executes on **Core 0**.
- `ProducerTaskB` executes on **Core 1**.
- `ConsumerTask` executes on **Core 0**.

3. Both Producer tasks shall send integer values to the **same Queue**.

4. Initialize each Producer with a different counting sequence to distinguish the origin of each value.

For example:

- `ProducerTaskA` starts at **100**
- `ProducerTaskB` starts at **200**

5. Modify the Serial Monitor messages so that each Producer identifies itself.

Example:

```text
Producer A (Core 0) -> 100
Producer B (Core 1) -> 200
```

6. Keep the Consumer unchanged. It should continue receiving values from the same Queue.

7. Build and execute the application.

8. Observe the order in which values are received.

---

## Example Output

```text
Producer B (Core 1) -> 211
Consumer (Core 0) <- 211

Producer A (Core 0) -> 111
Consumer (Core 0) <- 111

Producer B (Core 1) -> 212
Consumer (Core 0) <- 212

Producer A (Core 0) -> 112
Consumer (Core 0) <- 112
```

Notice that both Producer tasks successfully transmit data through the same Queue while the Consumer receives each value correctly.

> [!NOTE]
>
> Multiple tasks may safely send data to the same FreeRTOS Queue. The FreeRTOS SMP kernel guarantees that each transmitted element is inserted atomically, preventing data corruption even when multiple processor cores access the Queue concurrently.

---

## Expected Behavior

After running the application:

- Producer A always executes on **Core 0**.
- Producer B always executes on **Core 1**.
- The Consumer executes on **Core 0**.
- Both Producers safely share the same Queue.
- The Consumer receives values generated by both Producers.
- Values are received in the same order they are inserted into the Queue (FIFO).
- The insertion order depends on the scheduler and may vary between executions.

# Activity 4 - Event Notification with a Binary Semaphore

---

## Objective

Starting from the application developed in **Activity 3**, simplify the design by using a single Producer and a single Consumer.

Instead of transferring data through a Queue, the Producer will only notify the Consumer that an event has occurred.

The objective of this activity is to discover that a **Binary Semaphore** is the appropriate synchronization mechanism when tasks need to signal events without transferring data.

---

## Procedure

Continue using the application developed in **Activity 3**.

1. Simplify the application by removing **ProducerTaskB** and its associated task creation and affinity configuration.

2. Remove the Queue and all Queue-related code from the application.

3. Configure task affinity as follows:

- `ProducerTask` executes on **Core 0**.
- `ConsumerTask` executes on **Core 1**.

4. In this activity, assume that the Producer represents a device capable of generating an event.

For simplicity, the event will be **simulated every 500 ms** using `vTaskDelay()`.

5. When the simulated event occurs, the Producer shall notify the Consumer.

Based on the concepts introduced in **Chapter 2**, determine the appropriate FreeRTOS synchronization mechanism.

> **Hint:** Use a **Binary Semaphore**.

6. The Consumer shall remain blocked while waiting for the event notification.

When the notification is received, the Consumer shall continue its execution and indicate that the event has been handled.

7. Build and execute the application.

---

## Example Output

```text
Producer (Core 0) -> Event
Consumer (Core 1) <- Event Received

Producer (Core 0) -> Event
Consumer (Core 1) <- Event Received

Producer (Core 0) -> Event
Consumer (Core 1) <- Event Received
```

Notice that no application data is transferred between the tasks. The Binary Semaphore simply synchronizes the execution of both tasks.

> [!NOTE]
>
> In this laboratory, the event is simulated periodically by the Producer.
>
> - The simulated event occurs every 500 ms.
> - `xSemaphoreGive()` signals that the event has occurred.
> - `xSemaphoreTake()` detects the event and allows the Consumer to continue executing.
> - The `printf()` statements are used only to visualize the synchronization and are **not** part of the event itself.

---

## Expected Behavior

After running the application:

- The Producer always executes on **Core 0**.
- The Consumer always executes on **Core 1**.
- The Producer periodically generates a simulated event.
- The Consumer remains blocked until the event is signaled.
- Each generated event wakes the Consumer exactly once.
- No application data is transferred between the tasks.

# Activity 5 - Protecting a Shared Resource with a Mutex

---

## Objective

In the previous activity, the Producer and Consumer tasks synchronized their execution using a Binary Semaphore.

In this activity, two tasks will access a **shared resource** concurrently. The objective is to observe how concurrent access can produce inconsistent data and how a **Mutex** guarantees exclusive access to the shared resource.

---

## Procedure

Continue using the application developed in **Activity 4**.

### Part A - Observing a Race Condition

1. Remove the Binary Semaphore and all semaphore-related code from the application.

2. Create the following shared data structure.

```c
typedef struct
{
    uint32_t value;
    uint32_t checksum;
} SharedData_t;
```

3. Create two new tasks:

- `WriterTask`
- `ReaderTask`

4. Configure task affinity as follows:

- `WriterTask` executes on **Core 0**.
- `ReaderTask` executes on **Core 1**.

5. The `WriterTask` shall continuously update the shared structure.

To make the race condition easier to observe, intentionally insert a short delay between updating both fields.

6. The `ReaderTask` shall continuously verify that the following relationship is always satisfied:

```text
checksum = value × 2
```

Whenever the relationship is violated, display the inconsistent data.

7. Build and execute the application.

---

## Example Output (Without Mutex)

```text
Inconsistent Shared Data!

Value    : 4300
Checksum : 8598
Expected : 8600

Inconsistent Shared Data!

Value    : 4301
Checksum : 8600
Expected : 8602
```

Notice that the Reader occasionally observes the shared structure while it is only partially updated.

This situation is known as a **race condition**.

> [!NOTE]
>
> The delay inserted inside the WriterTask intentionally enlarges the inconsistent update window.
>
> This delay is included only for educational purposes to make the race condition easier to observe. In a real embedded application, critical sections should be kept as short as possible.

---

## Part B - Protecting the Shared Resource

Modify the application by protecting the shared structure with a **Mutex**.

Based on the concepts introduced in **Chapter 2**, determine the appropriate FreeRTOS synchronization mechanism.

> **Hint:** Use a **Mutex** (`xSemaphoreCreateMutex()`).

Both the `WriterTask` and the `ReaderTask` must acquire the Mutex before accessing the shared structure and release it immediately after completing their operation.

Build and execute the application again.

---

## Example Output (With Mutex)

```text
Shared Data is consistent.

Value    : 4300
Checksum : 8600

Shared Data is consistent.

Value    : 4301
Checksum : 8602

Shared Data is consistent.

Value    : 4302
Checksum : 8604
```

Notice that the Reader never observes an inconsistent state after the shared resource is protected.

---

## Expected Behavior

### Without Mutex

- The Writer executes on **Core 0**.
- The Reader executes on **Core 1**.
- The Reader occasionally detects inconsistent shared data.
- Multiple race conditions are observed.

### With Mutex

- Both tasks continue executing on different processor cores.
- The shared resource is accessed exclusively.
- No inconsistent shared data is observed.
- Every reported value satisfies:

```text
checksum = value × 2
```

## Key Concept

A **Mutex** protects a **critical section**, not an individual variable.

The goal is to guarantee that a complete operation on a shared resource is executed atomically, preventing other tasks from observing intermediate or inconsistent states.
