# Activity 1 - Sensor Measurements Without Inter-Task Communication

---

## Objective

In this activity, a **SensorTask** periodically acquires measurements while a **ControllerTask** attempts to use them. Since no communication mechanism exists between the tasks, the Controller cannot access the measurements produced by the Sensor.

The objective is to identify the need for an inter-task communication mechanism.

---

## Estimated Time

15 minutes

---

## Procedure

1. Open the **Template Project**.

2. Create two FreeRTOS tasks named:

   - `SensorTask`
   - `ControllerTask`

3. Configure both tasks with:

   - The same priority.
   - An infinite execution loop.
   - A delay of **1000 ms**.

4. Configure task affinity so that:

   - `SensorTask` always executes on **Core 0**.
   - `ControllerTask` always executes on **Core 1**.

5. The `SensorTask` should periodically print an incrementing integer value.

6. The `ControllerTask` should periodically display a placeholder.

The question marks represent a value that the Controller cannot obtain because no communication mechanism has been implemented yet.

7. Do **not** use any communication mechanism.

Do **not** use:

- Queue
- Binary Semaphore
- Mutex
- Event Group

8. Start the scheduler.

---

## Example Output

```text
Sensor (Core 0) -> Temperature = 0 °C
Controller (Core 1) <- Temperature = ???

Sensor (Core 0) -> Temperature = 1 °C
Controller (Core 1) <- Temperature = ???

Sensor (Core 0) -> Temperature = 2 °C
Controller (Core 1) <- Temperature = ???
```

Notice that the Sensor continuously generates values while the Controller has no mechanism to obtain them.

---

## Expected Behavior

After running the application:

- The Sensor always executes on **Core 0**.
- The Controller always executes on **Core 1**.
- The Sensor continuously generates new values.
- The Controller periodically displays the placeholder `???`.
- The Controller cannot obtain the values generated by the Sensor.
- Both tasks execute independently because no communication mechanism exists between them.

# Activity 2 - Transferring Sensor Measurements with a Queue

---

## Objective

Modify the application developed in **Activity 1** so that the **Controller**, executing on **Core 1**, is able to display the values generated by the **Sensor**, executing on **Core 0**.

The objective of this activity is to discover how a communication mechanism enables tasks running on different processor cores to exchange information safely.

---

## Procedure

Continue using the application developed in **Activity 1**.

1. The Sensor already generates an incrementing temperature value every 1000 ms.

2. The Controller currently displays:

```text
Controller (Core 1) <- Temperature = ???
```

Your objective is to replace the question marks with the values generated by the Sensor.

3. Based on the concepts introduced in **Chapter 2**, identify the appropriate FreeRTOS communication mechanism.

> **Hint:** Use a **FreeRTOS Queue**.

4. Create a Queue capable of storing integer values.

5. Modify the `SensorTask` so that every generated value is sent to the Queue.

6. Modify the `ControllerTask` so that it waits for a value from the Queue and displays it after reception. Remove the delay from the task.

7. Keep the same task affinity configuration:

- `SensorTask` executes on **Core 0**.
- `ControllerTask` executes on **Core 1**.

8. Build and execute the application.

---

## Example Output

```text
Sensor (Core 0) -> Temperature = 0 °C
Controller (Core 1) <- Received Temperature = 0 °C

Sensor (Core 0) -> Temperature = 1 °C
Controller (Core 1) <- Received Temperature = 1 °C

Sensor (Core 0) -> Temperature = 1 °C
Controller (Core 1) <- Received Temperature = 1 °C
```

Notice that the values generated on **Core 0** are successfully received by the task executing on **Core 1**.

> [!NOTE]
>
> A FreeRTOS Queue behaves identically whether the communicating tasks execute on the same processor core or on different processor cores. The FreeRTOS SMP kernel automatically guarantees safe concurrent access to the Queue.

---

## Expected Behavior

After running the application:

- The Sensor always executes on **Core 0**.
- The Controller always executes on **Core 1**.
- Every value generated by the Sensor is eventually received by the Controller.
- The Sensor and Controller do not necessarily alternate their execution.
- The Queue preserves the order of the transmitted values (FIFO).

# Activity 3 - Multiple Sensors Sharing a Queue

---

## Objective

Extend the application developed in **Activity 2** by introducing a second Sensor task.

The objective of this activity is to observe how multiple tasks executing on different processor cores can safely share the same FreeRTOS Queue while a single Controller receives all transmitted data.

---

## Procedure

Continue using the application developed in **Activity 2**.

1. Create the following data structures.

```c
typedef enum
{
    SENSOR_TEMPERATURE,
    SENSOR_DISTANCE
} SensorType_t;

typedef struct
{
    SensorType_t sensor;
    uint32_t value;
} SensorMessage_t;
```

2. Rename the existing `SensorTask` to `TemperatureTask`.

3. Create a second task named `DistanceTask`.

4. Configure task affinity as follows:

- `TemperatureTask` executes on **Core 0**.
- `DistanceTask` executes on **Core 1**.
- `ControllerTask` executes on **Core 0**.

5. Configure different sampling periods for each sensor.

- `TemperatureTask` samples every **1000 ms**.
- `DistanceTask` samples every **500 ms**.

6. Both sensor tasks shall send `SensorMessage_t` structures through the **same Queue**.

7. Initialize each sensor with a different counting sequence to distinguish the origin of each measurement.

For example:

- `TemperatureTask` starts at **20 °C**
- `DistanceTask` starts at **100 cm**

8. Modify the `ControllerTask` so that it identifies the sensor that generated each received measurement and displays an appropriate message in the Serial Monitor.

9. Build and execute the application.

10. Observe the order in which measurements are received by the Controller.

---

## Example Output

```text
Temperature Task (Core 0) -> Temperature = 20 °C
Controller (Core 0) <- Received Temperature = 20 °C

Distance Task (Core 1) -> Distance = 100 cm
Controller (Core 0) <- Received Distance = 100 cm

Distance Task (Core 1) -> Distance = 101 cm
Controller (Core 0) <- Received Distance = 101 cm

Temperature Task (Core 0) -> Temperature = 21 °C
Controller (Core 0) <- Received Temperature = 21 °C

Distance Task (Core 1) -> Distance = 102 cm
Controller (Core 0) <- Received Distance = 102 cm
```

Notice that both Sensor tasks successfully transmit data through the same Queue while the Controller receives each value correctly.

> [!NOTE]
>
> Multiple tasks may safely send data to the same FreeRTOS Queue. The FreeRTOS SMP kernel guarantees that each transmitted element is inserted atomically, preventing data corruption even when multiple processor cores access the Queue concurrently.

---

## Expected Behavior

After running the application:

- Temperature Task always executes on **Core 0**.
- Distance Task always executes on **Core 1**.
- The Controller executes on **Core 0**.
- Both Sensor safely share the same Queue.
- The Controller receives values generated by both Sensors.
- Values are received in the same order they are inserted into the Queue (FIFO).
- The insertion order depends on the scheduler and may vary between executions.

# Activity 4 - Event Notification with a Binary Semaphore

---

## Objective

Starting from the application developed in **Activity 3**, simplify the system by using a single **SensorTask** and a single **ControllerTask**.

Instead of transferring sensor measurements through a Queue, the SensorTask will only notify the ControllerTask that a new measurement is available.

The objective of this activity is to discover that a **Binary Semaphore** is the appropriate synchronization mechanism when tasks need to signal events without transferring application data.

---

## Estimated Time

20 minutes

---

## Procedure

Continue using the application developed in **Activity 3**.

1. Remove the `DistanceTask` and all associated task creation and affinity configuration.

2. Remove the Queue and all Queue-related code from the application.

3. Configure task affinity as follows:

- `SensorTask` executes on **Core 0**.
- `ControllerTask` executes on **Core 1**.

4. Assume that the SensorTask represents a hardware device capable of generating a **Measurement Ready** event.

For simplicity, the event will be simulated every **500 ms** using `vTaskDelay()`.

5. Whenever a new measurement becomes available, the SensorTask shall notify the ControllerTask.

Based on the concepts introduced in **Chapter 2**, determine the appropriate FreeRTOS synchronization mechanism.

> **Hint:** Use a **Binary Semaphore**.

6. The ControllerTask shall remain blocked while waiting for the measurement-ready notification.

When the notification is received, the ControllerTask shall continue executing and indicate that the measurement has been processed.

7. Build and execute the application.

---

## Example Output

```text
Sensor Task (Core 0) -> Measurement Ready
Controller (Core 1) <- Measurement Processed

Sensor Task (Core 0) -> Measurement Ready
Controller (Core 1) <- Measurement Processed

Sensor Task (Core 0) -> Measurement Ready
Controller (Core 1) <- Measurement Processed
```

Notice that no sensor data is transferred between the tasks. The Binary Semaphore simply synchronizes the execution of both tasks.

> [!NOTE]
>
> In this activity, the measurement-ready event is simulated periodically by the SensorTask.
>
> - A new measurement becomes available every **500 ms**.
> - `xSemaphoreGive()` signals that the measurement is ready.
> - `xSemaphoreTake()` allows the ControllerTask to detect the event and continue executing.
> - The `printf()` statements are used only to visualize the synchronization and are **not** part of the event itself.

---

## Expected Behavior

After running the application:

- The SensorTask always executes on **Core 0**.
- The ControllerTask always executes on **Core 1**.
- The SensorTask periodically generates a **Measurement Ready** event.
- The ControllerTask remains blocked until the event is signaled.
- Each generated event wakes the ControllerTask exactly once.
- No application data is transferred between the tasks.

# Activity 5 - Protecting a Shared Resource with a Mutex

---

## Objective

In the previous activity, the Producer and Consumer tasks synchronized their execution using a Binary Semaphore.

In this activity, two tasks will access a **shared resource** concurrently. The objective is to observe how concurrent access can produce inconsistent data and how a **Mutex** guarantees exclusive access to the shared resource.

---

## Procedure

Continue using the application developed in **Activity 4**.

### Part A - Observing a Race Condition

1. Remove the Binary Semaphore and all semaphore-related code from the application.

2. Create the following shared data structure.

```c
typedef struct
{
    uint32_t value;
    uint32_t checksum;
} SharedData_t;
```

3. Create two new tasks:

- `WriterTask`
- `ReaderTask`

4. Configure task affinity as follows:

- `WriterTask` executes on **Core 0**.
- `ReaderTask` executes on **Core 1**.

5. The `WriterTask` shall continuously update the shared structure.

To make the race condition easier to observe, intentionally insert a short delay between updating both fields.

6. The `ReaderTask` shall continuously verify that the following relationship is always satisfied:

```text
checksum = value × 2
```

Whenever the relationship is violated, display the inconsistent data.

7. Build and execute the application.

---

## Example Output (Without Mutex)

```text
Inconsistent Shared Data!

Value    : 4300
Checksum : 8598
Expected : 8600

Inconsistent Shared Data!

Value    : 4301
Checksum : 8600
Expected : 8602
```

Notice that the Reader occasionally observes the shared structure while it is only partially updated.

This situation is known as a **race condition**.

> [!NOTE]
>
> The delay inserted inside the WriterTask intentionally enlarges the inconsistent update window.
>
> This delay is included only for educational purposes to make the race condition easier to observe. In a real embedded application, critical sections should be kept as short as possible.

---

## Part B - Protecting the Shared Resource

Modify the application by protecting the shared structure with a **Mutex**.

Based on the concepts introduced in **Chapter 2**, determine the appropriate FreeRTOS synchronization mechanism.

> **Hint:** Use a **Mutex** (`xSemaphoreCreateMutex()`).

Both the `WriterTask` and the `ReaderTask` must acquire the Mutex before accessing the shared structure and release it immediately after completing their operation.

Build and execute the application again.

---

## Example Output (With Mutex)

```text
Shared Data is consistent.

Value    : 4300
Checksum : 8600

Shared Data is consistent.

Value    : 4301
Checksum : 8602

Shared Data is consistent.

Value    : 4302
Checksum : 8604
```

Notice that the Reader never observes an inconsistent state after the shared resource is protected.

---

## Expected Behavior

### Without Mutex

- The Writer executes on **Core 0**.
- The Reader executes on **Core 1**.
- The Reader occasionally detects inconsistent shared data.
- Multiple race conditions are observed.

### With Mutex

- Both tasks continue executing on different processor cores.
- The shared resource is accessed exclusively.
- No inconsistent shared data is observed.
- Every reported value satisfies:

```text
checksum = value × 2
```

## Key Concept

A **Mutex** protects a **critical section**, not an individual variable.

The goal is to guarantee that a complete operation on a shared resource is executed atomically, preventing other tasks from observing intermediate or inconsistent states.

# Activity 6 - Coordinating Multiple Events with an Event Group

---

## Objective

In the previous activities, you learned how to:

- Transfer data using a Queue.
- Synchronize two tasks using a Binary Semaphore.
- Protect a shared resource using a Mutex.

In this activity, you will coordinate **multiple independent events** using an **Event Group**.

The objective is to observe how a task can wait for **any** event or **all** events before continuing its execution.

---

## Procedure

Continue using the application developed in **Activity 5**.

### Part A - Waiting for Any Event (OR)

1. Remove the Mutex and all mutex-related code from the application.

2. Create an **Event Group**.

3. Define the following event bits.

```c
#define SENSOR_READY    (1 << 0)
#define UART_READY      (1 << 1)
```

4. Create three tasks:

- `SensorTask`
- `CommunicationTask`
- `ControllerTask`

5. Configure task affinity as follows:

- `SensorTask` executes on **Core 0**.
- `CommunicationTask` executes on **Core 1**.
- `ControllerTask` has no core affinity restriction.

6. The `SensorTask` shall periodically indicate that a sensor measurement is available by setting the `SENSOR_READY` bit.

7. The `CommunicationTask` shall periodically indicate that a communication channel is ready by setting the `UART_READY` bit.

8. Configure the `ControllerTask` to wait until **either** event occurs.

> **Hint:** Configure `xEventGroupWaitBits()` to wait for **any** event.

9. Build and execute the application.

---

## Example Output (Waiting for Any Event)

```text

Controller (Core 0): Event Received -> UART

Sensor Task (Core 0): Sensor Ready
Controller (Core 1): Event Received -> Sensor

Sensor Task (Core 0): Sensor Ready
Controller (Core 0): Event Received -> Sensor

Communication Task (Core 1): UART Ready
Controller (Core 1): Event Received -> UART
```

Notice that the Controller resumes execution whenever **either** event occurs.

---

## Part B - Waiting for All Events (AND)

Modify only the configuration of `xEventGroupWaitBits()` so that the `ControllerTask` waits until **both** events have occurred.

Build and execute the application again.

---

## Example Output (Waiting for All Events)

```text
Sensor Task (Core 0): Sensor Ready
Sensor Task (Core 0): Sensor Ready
Sensor Task (Core 0): Sensor Ready

Communication Task (Core 1): UART Ready

Controller (Core 0): Event Received -> Sensor UART

Sensor Task (Core 0): Sensor Ready
Sensor Task (Core 0): Sensor Ready

Communication Task (Core 1): UART Ready

Controller (Core 1): Event Received -> Sensor UART
```

Notice that the Controller executes only after **both** events are available.

> [!NOTE]
>
> An Event Group stores the **state** of each event bit, not the number of times the event occurred.
>
> If the same event is generated multiple times before the waiting task clears the bit, the Event Group still indicates only that the event has occurred.

- Multiple Sensor events do not cause multiple Controller executions.

# Laboratory 2 Summary

Throughout this laboratory, you explored the primary synchronization mechanisms provided by FreeRTOS SMP.

| Requirement | FreeRTOS Mechanism |
|-------------|--------------------|
| Transfer data between tasks | Queue |
| Notify that an event occurred | Binary Semaphore |
| Protect a shared resource | Mutex |
| Wait for multiple synchronization conditions | Event Group |

Selecting the appropriate synchronization mechanism is an essential engineering decision when designing multicore embedded systems. Each mechanism addresses a different coordination problem, and choosing the correct one improves system correctness, readability, and maintainability.
